Himafy 修正指示書（PM Directive）

発行: 2026/03/07 | 対象: 企画書v6との乖離解消（課金機能を除く）| 想定期間: 3スプリント（約3週間）

PM判断：なぜこの順番で切り込むのか

企画書との乖離は10件以上ありますが、すべてを同時に直すのではなく「ユーザーが触った瞬間に感じる価値」を基準に優先順位をつけました。

Himafyの核心的価値は「AIが"実行可能な"予定を提案してくれること」です。現状のサイトでは提案が抽象的すぎて、ユーザーが「これは自分の暇な時間に本当にできそう」と感じられません。ここが崩れていると、どれだけ機能を足しても定着しません。

したがって、Sprint 1では「提案の質」と「フィードバックループ」に全振りします。位置情報やPWAは重要ですが、まず提案の骨格が企画書の想定レベルに達しないと、その上に何を積んでも意味がありません。

Sprint 2で「位置情報×天気」を入れることで企画書のコア差別化を実現し、Sprint 3でPWA・通知・UX改善で仕上げます。

SPRINT 1 提案の骨格を企画書レベルに引き上げる 1週間 | 4タスク
T-1 AI提案プロンプトの全面改修 乖離: 06 AIパーソナライゼーション Impact: 最大 3日

企画書との乖離

企画書の提案出力例：「17:30〜19:00 自宅近くの○○カフェで読書」「出張先の名古屋で空き時間にご当地グルメ」
現状の提案：「最新業界レポートの深掘り分析（約90分）」— 時間帯なし、場所なし、具体的アクションなし

実装指示

  • システムプロンプトの再設計：以下の構造で提案を出力させる
  • 出力フォーマットを JSON で厳密に定義（タイトル / ステップ配列 / 場所 / 所要時間 / カテゴリタグ）
  • Google Calendar の空き時間情報をコンテキストに含める（T-2と連動）
  • ユーザーの選択タグ全量をプロンプトに注入し、組み合わせで提案をパーソナライズ
  • 3候補生成時に「インドア/アウトドア/学習」など異なるタイプを混ぜるルールを追加
// 改修後のシステムプロンプト骨格 const systemPrompt = ` あなたはHimafyの予定提案AIです。 ユーザーの空き時間に合った、"今日すぐ実行できる"具体的な予定を3つ提案してください。 ## 入力情報 - ユーザータグ: ${userTags.join(', ')} - カテゴリ: ${selectedCategory} - 空き時間: ${freeSlots.map(s => s.start + '〜' + s.end).join(', ')} - 居住地: ${user.location || '未設定'} - 天気: ${weather?.description || '不明'} - 過去の高評価タグ: ${topRatedTags.join(', ')} ## 出力ルール（厳守） 各提案は以下のJSON形式で返してください: { "title": "動詞で始まる具体的なタイトル（20字以内）", "description": "なぜこのユーザーにおすすめかの理由（50字以内）", "steps": [ { "action": "具体的なアクション", "duration": "15分" } ], "location": "場所名（自宅/カフェ名/公園名/オンライン）", "timeSlot": "14:00〜15:30", "totalMinutes": 90, "relatedTags": ["追加候補のタグ"] } ## 禁止事項 - 「〜について学ぶ」のような抽象的な提案 - 空き時間を超える所要時間の提案 - 3つの提案が同じタイプ（例:全部読書）にならないこと `;

受け入れ基準

  • 提案に「具体的な時間帯」が含まれている（例: 14:00〜15:30）
  • 提案に「場所」が含まれている（例: 自宅、最寄りのカフェ）
  • 提案に2〜3個の「具体的ステップ」が含まれている
  • 3候補のうち少なくとも2つは異なるタイプの提案である
  • 空き時間を超える提案が生成されないこと
T-2 Google Calendar空き時間の提案への反映 乖離: 06 空き時間→スロット自動算出 1.5日

企画書との乖離

企画書「空き時間 → 勤務時間外のスロットを自動算出」。Google Calendar連携は実装済みだが、提案生成時に空き時間情報が活用されていない。提案に具体的な時間帯が含まれていない。

実装指示

  • Calendar APIから当日〜翌日の予定を取得し、空きスロットを計算するユーティリティを作成
  • 最短30分以上の空きスロットを抽出し、AI提案のコンテキストに注入
  • Calendar未連携ユーザーには「空き時間を手動で入力」するフォールバックUIを提供
  • 提案選択（「この予定にする」）時にCalendarに予定を実際に書き込むAPIを実装
// lib/calendar.ts - 空き時間算出ロジック export async function getFreeSlots(userId: string, date: Date) { const events = await getCalendarEvents(userId, date); const dayStart = setHours(date, 8); // 活動開始 8:00 const dayEnd = setHours(date, 23); // 活動終了 23:00 const slots = calculateGaps(events, dayStart, dayEnd); return slots.filter(s => s.durationMinutes >= 30); // 30分以上のみ } // 予定をCalendarに書き込み export async function addToCalendar(userId: string, suggestion: Suggestion) { return await calendar.events.insert({ calendarId: 'primary', requestBody: { summary: suggestion.title, description: suggestion.steps.map(s => `- ${s.action} (${s.duration})`).join('\n'), start: { dateTime: suggestion.startTime }, end: { dateTime: suggestion.endTime }, } }); }

受け入れ基準

  • Calendar連携済みユーザーの提案に、実際の空き時間帯が反映されている
  • 「この予定にする」でGoogle Calendarに予定が書き込まれる
  • 既存予定との重複が発生しないこと
  • Calendar未連携ユーザーにもフォールバックがあること
T-3 予定選択後の「タグ追加」フロー実装 乖離: 04 主要機能「予定からタグ追加」 乖離: 05 ユーザーフロー Step 5 Impact: 大 2日

企画書との乖離

企画書の主要機能「予定からタグ追加：予定を選択した後、関連するタグをプロフィールに追加登録可能。使うほどプロフィールが充実」
企画書のユーザーフロー Step 5「タグ追加 + FB：予定から新タグ追加＆実行後にフィードバック」
→ 現状、この機能が完全に欠落。フィードバックループが断絶している。

実装指示

  • 「この予定にする」押下後に、確認画面 → タグ追加提案画面 を挟む
  • AI提案のレスポンスに含まれる relatedTags を表示し、ユーザーが追加したいものをタップで選択
  • 選択されたタグはプロフィールに自動追加される
  • スキップも可能にする（毎回強制しない）
  • 追加されたタグは設定画面のタグ一覧に反映
// 予定確定 → タグ追加 → 完了のフロー // Step 1: ユーザーが「この予定にする」をタップ const handleConfirm = async (suggestion) => { await addSchedule(suggestion); // DBに保存 await addToCalendar(user.id, suggestion); // Calendar書き込み // Step 2: タグ追加提案画面を表示 if (suggestion.relatedTags?.length > 0) { setShowTagSuggestion(true); setSuggestedTags(suggestion.relatedTags); } else { finishFlow(); } }; // Step 3: タグ追加画面のUI <TagSuggestionSheet> <h3>関連タグを追加しますか？</h3> <p>選んだ予定に関連するタグです。追加すると次回の提案がより精確になります</p> {suggestedTags.map(tag => ( <TagChip key={tag} onToggle={toggleTag} /> ))} <Button>追加して完了</Button> <button>スキップ</button> </TagSuggestionSheet>

受け入れ基準

  • 予定確定後にタグ追加提案が表示される
  • タグをタップするとプロフィールに追加される
  • 設定画面のタグ一覧に反映される
  • スキップ可能であること
  • 追加されたタグが次回のAI提案コンテキストに含まれること
T-4 フィードバック学習の可視化と提案反映 乖離: 04 パーソナライズ学習 1.5日

企画書との乖離

企画書「蓄積されたフィードバックとタグ履歴をもとにAIを継続チューニングし、提案の質を進化」。フィードバック画面（5段階評価）は存在するが、それが次回提案に反映される仕組みがなく、ユーザーにも「学習している」ことが伝わらない。

実装指示

  • バックエンド：フィードバック結果（評価+コメント+対象提案のタグ）をDBに蓄積
  • プロンプト注入：直近10件の高評価/低評価タグ・提案傾向をシステムプロンプトに追加（T-1と連動）
  • UI：ダッシュボードに「AIの理解度」ウィジェットを追加（例:「あなたの好みを12件のフィードバックから学習中」）
  • フィードバック送信後に「ありがとうございます。次回の提案に反映します」トーストを表示
// AI提案時にフィードバック履歴をコンテキスト注入 const feedbackContext = await buildFeedbackContext(userId); // feedbackContext の中身（例） { positivePatterns: ["カフェ系の提案が高評価", "90分以上は低評価傾向"], recentHighRated: ["読書", "カフェ巡り", "プログラミング"], recentLowRated: ["掃除系", "長時間の学習"], totalFeedbacks: 12 } // プロンプトに追記 const fbPrompt = ` ## ユーザーの好み傾向（フィードバック ${feedbackContext.totalFeedbacks}件から学習） - 高評価: ${feedbackContext.recentHighRated.join(', ')} - 低評価: ${feedbackContext.recentLowRated.join(', ')} - 傾向: ${feedbackContext.positivePatterns.join(', ')} → 高評価タグに近い提案を優先し、低評価傾向は避けてください。 `;

受け入れ基準

  • 高評価した提案に近いジャンルが次回提案で優先される
  • 低評価した傾向の提案が減少する
  • ダッシュボードに学習状況が可視化されている
  • FB送信後のトースト表示がある

T-6 天気API連携 乖離: 06 天気情報→屋内/屋外提案 技術スタック: Weather API 1日

実装指示

  • OpenWeatherMap API（Free tier: 1,000コール/日）を利用
  • ユーザーの居住地座標から当日〜翌日の天気を取得
  • Redisに天気データを30分間キャッシュ（API節約）
  • 天気情報をAI提案のコンテキストに注入（雨→屋内提案優先 等）
// lib/weather.ts export async function getWeather(lat: number, lng: number) { const cacheKey = `weather:${lat.toFixed(2)}:${lng.toFixed(2)}`; const cached = await redis.get(cacheKey); if (cached) return JSON.parse(cached); const res = await fetch( `https://api.openweathermap.org/data/2.5/forecast?lat=${lat}&lon=${lng}&appid=${API_KEY}&lang=ja&units=metric` ); const data = await res.json(); await redis.setex(cacheKey, 1800, JSON.stringify(data)); // 30分キャッシュ return data; }
T-7 提案への位置情報・天気の統合 乖離: 02 「居住地×位置×時間」 Impact: 大（差別化の核心） 2日

企画書との乖離

企画書のコンセプト3本柱の1つ「居住地 × 位置 × 時間 — 居住地・現在地と空き時間を考慮した実行可能な予定のみ提案」が丸ごと未実装。

実装指示

  • T-5の居住地情報とT-6の天気データをT-1のプロンプトに統合
  • 居住地周辺の具体的な施設名（公園、カフェ、図書館等）を Google Places API で取得し、提案に組み込む
  • 天気に応じた提案ルール: 雨天→屋内系優先、晴天→屋外系も混ぜる
  • 提案カードのUIに「📍 場所名」「🌤 天気」を表示
  • 居住地未設定ユーザーには「オンライン」「自宅」中心の提案にフォールバック

リスクと対策

Places APIのコスト：Autocomplete は$2.83/1000リクエスト。初期段階は「市区町村＋ジャンル」のテンプレート提案（例:「○○市のカフェ」）で代替し、ユーザー数増加後にPlace Details（個別店舗名）へ段階的に移行する。

PMからの補足

Sprint 1が最も重要です。ここで「提案の具体性」と「フィードバックループ」が動き始めれば、企画書が描いている「使うたびにタグが増え、AIが学習し、提案精度が向上する」という好循環が初めて成立します。

Sprint 2の位置情報・天気は差別化要素ですが、Sprint 1の提案品質が低いままだと「位置情報を使って抽象的な提案をする」だけになってしまいます。順番を守ってください。

Sprint 3のPWA・Push通知はリテンション施策です。ユーザーに「また使いたい」と思ってもらえるだけの提案品質がSprint 1-2で実現できてから意味を持ちます。

計測指標：Sprint 1完了時点で「提案受け入れ率（"この予定にする"のCTR）」が現状から有意に上昇していることを確認してからSprint 2に進んでください。もし改善が見られなければ、T-1のプロンプトをさらにチューニングするサブスプリントを挟みます。
